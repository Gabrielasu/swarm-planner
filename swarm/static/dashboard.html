Swarm Dashboard Guide

What is Swarm?

Swarm is a multi-agent planning orchestrator. It takes a project description and runs it through a pipeline of specialized LLM agents to produce an implementation-ready software specification -- a stateful task graph with self-contained prompt packets that coding agents can pick up and execute.

The pipeline outputs are written to .plan/ in your project directory. The primary output is graph.json (the task DAG) and tasks/*.json (self-contained prompt packets per task).

Pipeline Steps

The pipeline runs 11 sequential steps. Each step is a focused LLM agent with a specialized prompt:

1
Interviewer
Evaluates your input and asks clarifying questions if the brief is vague. Produces a structured brief with system purpose, user types, data entities, and constraints. Uses the Fast model tier (Haiku).
2
Codebase Analyzer
Scans existing code for brownfield projects. Only runs if --codebase was provided. Identifies existing patterns, tech stack, and integration points.
3
Decomposer
Breaks the system into components with clear ownership boundaries. Defines data flows between components. Includes a self-review step. Uses the Frontier model tier (Opus).
4
Contract Writer
Defines interface contracts at every component boundary -- function signatures, error cases, data schemas, and stub strategies for parallel development.
5
Contract Resolver
Makes contracts implementation-grade by resolving ambiguities, adding types, and ensuring consistency across boundaries. Processes contracts in batches of 3.
6
Adversary Loop
Attacks the plan looking for structural, data flow, and failure issues. Runs up to 3 rounds. Stops early if no critical/high findings remain. Uses the Frontier model tier (Opus).
7
Human Review
Pipeline pauses for you to review the architecture, contracts, and design decisions in .plan/review.md. Run swarm approve to continue.
8
Sequencer
Converts the architecture into an ordered, parallelizable task graph. Defines dependencies, file operations, and acceptance criteria for each task.
9
Simulator
Checks if each task can be one-shotted by a coding agent. Identifies gaps and missing information. Marks tasks as READY, NEEDS_REFINEMENT, or BLOCKED.
10
Refinement
Re-resolves contracts and tasks that failed the simulator's readiness check. Iterates until all tasks pass or a maximum iteration count is reached.
11
Graph Export
Builds the final graph.json and individual tasks/*.json prompt packets. Each packet is self-contained with inlined contracts.

Task Statuses

Tasks follow a state machine. Status transitions happen automatically when dependencies are completed or discoveries are reported.

○
Pending -- Dependencies not yet met. The task is blocked until upstream tasks are done.
▶
Ready -- All dependencies are done. This task can be picked up by a coding agent.
●
In Progress -- A coding agent is actively working on this task.
✓
Done -- Implementation complete. Downstream tasks that depended on this will become ready.
!
Invalidated -- Was done, but a discovery or upstream change broke it. Needs re-implementation.
↻
Needs Update -- The planner flagged this for revision due to a discovery. Cannot proceed until resolved.

Managing Tasks from the Dashboard

Click any task in the list or graph to open its detail drawer. From there you can:

Start Task
Marks a ready task as in-progress. Equivalent to swarm start <id>.
Complete Task
Marks an in-progress task as done. Cascades to unblock dependents. Equivalent to swarm done <id>.
Invalidate Task
Marks a task as invalidated (upstream change broke it). Cascades downstream. Equivalent to running invalidate_task().

Task Detail Drawer

When you click a task, the drawer shows the full prompt packet -- everything a coding agent receives:

Instruction
What to build -- the core description of the task.
Files
Files to create, modify, and read for context.
Contracts
Inlined interface contracts with function signatures, error shapes, and stubs.
Acceptance Criteria
Machine-verifiable assertions that define when the task is done.
Dependencies
Tasks that must be completed before this one (clickable links).
Unlocks
Downstream tasks that will become ready when this one is done.

Graph Visualization

The task graph shows the dependency DAG. Nodes are colored by status. Edges show dependencies with arrowheads pointing from upstream to downstream.

Pan
Click and drag anywhere on the graph to pan.
Zoom
Mouse wheel or trackpad scroll to zoom. Use +/- buttons or pinch gesture.
Fit
Click the circle button to fit all tasks into view.
Click
Click any node to open its detail drawer with full info and actions.

Discovery System

When a coding agent finds something during implementation that affects the plan, it records a discovery. Discoveries automatically flag affected tasks:

Types: missing_contract_fn, task_split_needed, dependency_missing, scope_change, blocker.

Affected tasks that were done become invalidated. Tasks that were pending/ready become needs_update. The graph version increments and the changelog records the mutation.
